## Что делает (поведение, не реализация)

- Входящее сообщение из Telegram больше не «оседает» в `inbox/` как мёртвый файл — система определяет, к какому проекту оно относится, и кладёт его туда, попутно строя связи в памяти (cross-references, concept stubs) через существующий `IngestPipeline`.
- Определение проекта — **по правилам, без LLM** (выбор пользователя):
  - R1: текст явно упоминает имя известного проекта vault → кладём в этот проект.
  - R2: заголовок/текст вида `ТЗ: <Name>`, `ТЗ <Name>`, `Проект называется <Name>`, `Project: <Name>` → распознаём как новый проект, создаём slug-папку.
  - R3: уверенного матча нет → остаётся в `inbox` (текущее поведение, не регрессия).
- Работает в двух режимах:
  - **Поток**: бот классифицирует каждое новое сообщение при приёме (`handle_message`).
  - **Разовый разбор**: CLI `process-inbox` разгребает уже накопленные файлы в `inbox/` (104 шт на момент создания спеки, включая ТЗ KeyStack).

## Кто использует / когда вызывается

- `telegram_bot.handle_message` — на каждое входящее без явного `@project`.
- `cli process-inbox` — вручную/по расписанию для накопленного inbox.

## Входные данные

| Параметр | Тип | Обязателен | Описание |
|----------|-----|-----------|----------|
| text | str | да | Тело сообщения/заметки |
| title | str | нет | Заголовок (если уже извлечён) |
| known_projects | list[str] | да | Список папок-проектов vault (`get_projects`) |

## Выходные данные / результат

- `RouteDecision{ project: str, note_type: str, reason: str, is_new_project: bool }`
- Побочно (через IngestPipeline): создан primary note в папке проекта, cross-refs у related заметок, concept stubs, запись в log.md.

## Edge cases — явно обработать

- [x] Пустой/слишком короткий текст → `inbox`, без создания проекта.
- [x] Несколько известных проектов упомянуты → берём проект с наибольшим числом совпадений; при ничьей → `inbox` (не угадываем).
- [x] Ложное срабатывание R2 на новый проект → создаём проект ТОЛЬКО при явном паттерне-заголовке, иначе `inbox`.
- [x] Slug нового проекта коллизирует с существующим → используем существующий проект (не дублируем).
- [x] `github-radar-*.md` и системные файлы в inbox → пропускать при `process-inbox`.
- [x] Файл уже содержит `project:` != inbox во frontmatter → не трогать.

## Что НЕ входит в эту фичу (scope boundary)

- ❌ LLM-классификация (отдельное решение, не сейчас).
- ❌ Авто-перемещение/переименование уже разнесённых заметок.
- ❌ Изменение логики голос/OCR/URL-обогащения в боте.
- ❌ LaunchAgent/расписание для process-inbox (запускается вручную; поток покрывает новые сообщения).
